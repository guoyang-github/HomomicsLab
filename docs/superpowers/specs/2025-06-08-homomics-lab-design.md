## 1. 概述与愿景

### 1.1 产品定位

HomomicsLab 是一款面向生命科学领域的通用 Agent 工具，旨在通过自然语言交互，帮助生物信息学研究者、湿实验科学家和临床研究人员完成从实验设计、数据分析到报告生成的全流程工作。

### 1.2 核心价值主张

- **降低生信分析门槛**：非专业用户可通过自然语言完成复杂分析
- **提升专业用户效率**：自动化重复工作，加速分析迭代
- **确保分析可复现**：完整记录分析血缘，支持一键复现
- **促进知识沉淀**：Skills 自进化，最佳实践持续积累

### 1.3 目标用户

| 用户类型 | 典型场景 | 核心需求 |
|---------|---------|---------|
| 湿实验科学家 | 测序数据下机后分析 | 无需编程，自然语言交互 |
| 生物信息学分析师 | 大规模单细胞/空间组学 | 自动化 pipeline、批量处理 |
| 临床研究员 | 临床样本分析、报告生成 | 合规、标准化、可审计 |
| 生物企业研发 | 药物筛选、靶点发现 | 可扩展、团队协作、知识管理 |

### 1.4 参考竞品

- **Biomni**: 实验辅助、数据分析、数据解读的闭环设计
- **AutoScientists**: 大型任务拆解、长程任务能力、高正确性
- **DeerFlow2**: Multi-agent 协作、任务调度
- **OmicVerse**: 生信生态集成、可视化策略

### 1.5 设计约束（已确认）

- **部署模式**: 混合模式（本地核心引擎 + 云端协作同步）
- **MVP 策略**: Agent 大脑优先——先做透任务拆解、Multi-agent、TODO 追踪、HITL
- **Multi-Agent**: 混合模式——日常中心化调度，复杂任务允许 Agent 间直接协作
- **交互形态**: 双轨并行——对话面板 + 可视化工作空间，实时联动
- **技术栈**: 无特定偏好，按模块选最优
- **工作流引擎**: Nextflow 为首选，Snakemake 为备选
- **协作模式**: 单用户为主，分享为静态报告
- **部署简化**: 本地支持 pip 安装，HPC 支持用户级安装，云端可选单服务器/Docker Compose，K8s 为未来选项

## 2. 架构方案

### 2.1 宏观架构选择：分层混合架构

采用"**内核紧凑 + 外延弹性**"的分层策略——核心 Agent 能力以模块化单体实现，外围能力以服务化插件形式存在。

```
┌──────────────────────────────────────────────────────┐
│              React 双轨前端 (TypeScript)              │
│        对话面板 ◄────实时同步────► 可视化工作空间       │
└──────────────────────┬───────────────────────────────┘
                       │ GraphQL + WebSocket
┌──────────────────────▼───────────────────────────────┐
│              API 网关 / BFF 层 (Node.js)              │
│         认证 │ 限流 │ 路由 │ 本地/云端模式适配         │
└──────────────────────┬───────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
┌───────▼──────┐ ┌────▼─────┐ ┌──────▼──────┐
│  Agent 核心   │ │  状态中心 │ │  项目管理   │
│  (Python)    │ │ (Python) │ │  (Python)   │
│              │ │          │ │             │
│ • 任务拆解   │ │ • 上下文 │ │ • 数据版本  │
│ • Multi-Agent│ │ • 记忆   │ │ • 协作权限  │
│ • TODO追踪   │ │ • 状态机 │ │ • 分享链接  │
│ • HITL      │ │          │ │             │
└───────┬──────┘ └────┬─────┘ └──────┬──────┘
        │             │              │
        └─────────────┴──────────────┘
                      │ 内部 API
        ┌─────────────┼─────────────┐
        │             │             │
┌───────▼─────┐ ┌────▼────┐ ┌──────▼──────┐
│ HPC 调度插件 │ │Skills   │ │ 可视化插件   │
│(Nextflow    │ │市场/注册│ │(Plotly/D3  │
│ + SLURM)    │ │中心      │ │ + 专用渲染) │
└─────────────┘ └─────────┘ └─────────────┘
        │                           │
        └────────── 消息队列 ─────────┘
```

### 2.2 分层原则

| 层次 | 设计哲学 | 技术选择 |
|------|----------|----------|
| **核心层**（Agent、状态、项目） | 单体但模块化，保证一致性 | Python/FastAPI，同进程内通信 |
| **网关层** | 统一出入口，屏蔽本地/云端差异 | Node.js/NestJS，支持 SSE/WebSocket |
| **插件层**（调度、Skills、可视化） | 服务化，独立演进 | 独立进程，HTTP/gRPC 接口 |
| **通信层** | 核心内部直接调用，跨层异步消息 | 内部：Python 调用；跨层：Redis（默认）/ NATS（高吞吐场景） |

### 2.3 架构优势

1. **符合 MVP 策略**：核心 Agent 能力可以快速集中开发，3-4个月可验证
2. **保留扩展空间**：HPC 调度、Skills 系统未来可以独立团队维护
3. **部署灵活**：本地部署时，插件可以内嵌为子进程（"紧凑模式"）；云端部署时，插件可独立容器化（"弹性模式"）
4. **双轨 UI 友好**：状态中心统一管理对话状态和工作空间状态，天然支持双向同步
5. **Multi-Agent 混合模式易于实现**：中心调度走内部 API（高效），Agent 间直接协作走消息队列（灵活）

## 3. 核心 Agent 引擎

### 3.1 总体架构

```
┌─────────────────────────────────────────────────────────────┐
```

### 3.2 Task Decomposer（任务拆解器）

参考 AutoScientists 和 Superpowers 的设计，采用**分层规划**策略：

**第一层：战略级规划（Strategic Planning）**
- 由"规划专用 Agent"（Planner Agent）执行
- 将高层目标拆解为**阶段（Phase）**
- 示例：单细胞分析 → [质控] → [降维聚类] → [注释] → [差异分析] → [可视化]

**第二层：战术级规划（Tactical Planning）**
- 每个阶段进一步拆解为**具体任务（Task）**
- 确定任务依赖关系（DAG）
- 识别需要 HITL 确认的关键决策点

**第三层：执行级规划（Execution Planning）**
- 将任务映射到具体 Skills/Tools
- 生成参数模板（待用户确认或自动填充）
- 评估资源需求（CPU/内存/时间）

```python
class TaskNode:
    id: str
    name: str
    description: str
    phase: str                    # 所属阶段
    status: TaskStatus            # pending/running/completed/failed/awaiting_human
    dependencies: List[str]       # 依赖的任务ID
    agent_assignment: str         # 分配的专业Agent
    skills_required: List[str]    # 所需Skills
    hitl_checkpoints: List[Checkpoint]  # HITL检查点
    estimated_duration: int       # 预估耗时（分钟）
    retry_policy: RetryPolicy     # 失败重试策略
```

### 3.3 Orchestrator（任务编排器）

采用**状态机驱动**的执行模型：

```
┌─────────┐    依赖满足     ┌─────────┐    完成      ┌─────────┐
│ PENDING │ ──────────────► │ RUNNING │ ──────────► │COMPLETED│
└────┬────┘                 └────┬────┘             └─────────┘
     │                           │
     │ 人工确认                   │ 失败
     ▼                           ▼
┌─────────┐                 ┌─────────┐    重试      ┌─────────┐
│AWAITING │                 │ FAILED  │ ──────────► │ RUNNING │
│ _HUMAN  │                 └────┬────┘  (未超限)    └─────────┘
└─────────┘                      │
                                 │ 重试超限
                                 ▼
                            ┌─────────┐
                            │ ABORTED │
                            └─────────┘
```

**调度策略**：
- **广度优先**：同一阶段的任务尽可能并行执行
- **资源感知**：根据当前 HPC 队列状态动态调整并发度
- **优先级抢占**：用户标记紧急的任务可插队

### 3.4 Multi-Agent 协作模型（混合模式）

**模式 1：中心调度（默认）**
- Orchestrator 将任务分配给最合适的专业 Agent
- Agent 完成任务后向 Orchestrator 汇报
- 适合：明确的、可独立执行的任务

**模式 2：Agent 间直接协作（复杂探索）**
- 触发条件：任务涉及多领域交叉、需要反复迭代
- 机制：Agent 间通过**共享上下文通道**直接通信
- 示例：
  ```
  AnalysisAgent: "发现细胞群注释存在矛盾，@ExperimentAgent 请确认样本分组是否正确"
  ExperimentAgent: "重新核查 Protocol，第3步的过滤参数有误，建议调整..."
  AnalysisAgent: "收到，重新执行质控步骤"
  ```

**Agent 类型设计**：

| Agent 类型 | 职责 | 触发条件 |
|-----------|------|---------|
| `PlannerAgent` | 任务拆解、资源评估 | 收到新请求时 |
| `BioinfoAgent` | 生信分析执行 | 涉及数据处理时 |
| `VizAgent` | 可视化生成 | 需要图表时 |
| `ExperimentAgent` | 实验设计/Protocol | 湿实验相关 |
| `QAAgent` | 结果验证、质控 | 关键节点自动触发 |
| `ReportAgent` | 报告生成 | 任务完成时 |

### 3.5 Human-in-the-Loop（HITL）机制

HITL 不是简单的"弹窗提问"，而是**融入工作流的智能决策点**：

**触发条件**：
1. **不确定性阈值**：LLM 对某个决策的置信度 < 0.7
2. **成本阈值**：某步操作消耗资源超过预设（如 >100 CPU·小时）
3. **风险阈值**：操作可能破坏数据（如删除、覆盖）
4. **策略阈值**：用户预先设定的必须确认环节（如发表级分析的最终参数）
5. **异常检测**：Agent 发现数据异常或结果矛盾

**交互形式**（双轨 UI 适配）：
- **对话侧**：以自然语言说明上下文和选项，引导用户决策
- **工作空间侧**：高亮相关数据/参数，提供可视化对比辅助决策

```python
class HITLCheckpoint:
    trigger_reason: HITLTrigger          # 触发原因
    context_summary: str                 # 上下文摘要（已压缩）
    options: List[Option]                # 可选项
    default_option: Optional[Option]     # 推荐选项（带解释）
    timeout: int                         # 超时时间（默认24h）
    escalation: EscalationPolicy         # 超时处理策略
```

### 3.6 上下文压缩策略

采用**分层记忆**架构：

```
┌─────────────────────────────────────────┐
│  工作记忆（Working Memory）               │
│  • 当前任务上下文（最近10轮对话）          │
│  • 当前任务节点状态                       │
│  • 常驻内存，LLM直接可见                  │
└─────────────────────────────────────────┘
           │ 自动摘要 & 归档
           ▼
┌─────────────────────────────────────────┐
│  短期记忆（Short-term Memory）            │
│  • 本 session 的历史摘要                  │
│  • 已完成的任务树快照                     │
│  • 向量存储，可检索                       │
└─────────────────────────────────────────┘
           │ 关键信息提取
           ▼
┌─────────────────────────────────────────┐
│  长期记忆（Long-term Memory）             │
│  • 用户偏好（常用参数、关注指标）          │
│  • 领域知识（常用分析策略）               │
│  • 项目级上下文（本项目数据特征）          │
└─────────────────────────────────────────┘
```

**智能压缩规则**：
- **相关性过滤**：基于任务目标，只保留相关的上下文
- **摘要生成**：长文本由专门的 Summarizer Agent 提炼关键结论
- **向量化检索**：历史上下文向量化存储，按需检索（RAG）
- **渐进式遗忘**：按时间衰减 + 重要性加权，自动清理低价值信息
- **用户显式标记"重要"**的内容永不自动丢弃
- **分析结果**保留结构化摘要，丢弃原始中间过程
- **错误和异常信息**保留完整（用于调试和审计）
- **参数设置**保留（用于复现）

## 4. 状态与上下文管理

### 4.1 总体设计哲学

双轨 UI 的核心挑战是**状态一致性**：对话面板中的操作必须立即反映在工作空间中，反之亦然。采用**单一事实源（Single Source of Truth）** + **响应式订阅**的架构。

### 4.2 状态分层架构

采用 **CRDT-inspired + 事件溯源** 的混合模式：

**L1：实时会话状态（Ephemeral State）**
- **生命周期**：用户 session 期间存在
- **内容**：当前对话上下文、工作空间视图状态（缩放、选中节点）
- **同步**：WebSocket 实时推送，延迟 < 100ms
- **存储**：Redis（本地模式用内存）

**L2：项目工作状态（Project State）**
- **生命周期**：项目存续期间
- **内容**：任务树、数据文件索引、分析参数、中间结果
- **同步**：写操作持久化到数据库 + 发布变更事件
- **存储**：PostgreSQL（结构化数据）+ 对象存储（文件）

**L3：长期知识状态（Knowledge State）**
- **生命周期**：跨项目、跨 session
- **内容**：用户偏好、领域知识、Skills 学习成果
- **同步**：异步更新，允许最终一致性
- **存储**：PostgreSQL + 向量数据库（Milvus 默认 / Pinecone 云端）

### 4.3 上下文压缩引擎

设计为**可插拔的压缩管道**：

```
┌─────────────────────────────────────────────────────────────┐
│                  Context Compression Pipeline                │
│                                                             │
│  输入：原始上下文（可能包含MB级别的日志、数据、对话）            │
│       │                                                     │
│       ▼                                                     │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐     │
│  │  Relevance  │───►│ Summarizer  │───►│  Semantic   │     │
│  │   Filter    │    │   Agent     │    │  Compressor │     │
│  │ （相关性过滤）│    │  （摘要生成） │    │ （语义压缩）  │     │
│  └─────────────┘    └─────────────┘    └─────────────┘     │
│       │                   │                   │              │
│       ▼                   ▼                   ▼              │
│  丢弃无关信息        生成结构化摘要        向量化存储可        │
│  （如：调试日志）    （保留关键结论）       检索的语义碎片      │
│                                                             │
│  输出：压缩后的上下文（保留关键信息，可检索历史细节）            │
└─────────────────────────────────────────────────────────────┘
```

**Relevance Filter（相关性过滤器）**：
- 基于任务目标动态计算每条上下文的相关性得分
- 多维度评分：语义相似度、时间邻近性、用户显式标记、结果依赖性、Agent 重要性报告

**Summarizer Agent（摘要生成器）**：
- 专用轻量级 Agent，负责将长上下文提炼为结构化摘要
- 输出：关键结论（3-5条）、关键参数、关键结果、警告、原始文本哈希、存储位置

**Semantic Compressor（语义压缩器）**：
- 将历史上下文向量化存储，支持按需检索
- 基于任务范围过滤 + 语义相似度检索

### 4.4 双轨同步机制

采用 **操作转换（Operational Transformation）** 思想的简化版：

```
┌──────────┐      Operation      ┌──────────┐
│  对话面板  │ ◄────────────────► │ 工作空间  │
└────┬─────┘   （用户意图抽象）    └────┬─────┘
     │                                 │
     └──────────┬──────────────────────┘
                ▼
        ┌───────────────┐
        │  Operation    │
        │  Interpreter  │
        │ （意图解析器）  │
        └───────┬───────┘
                ▼
        ┌───────────────┐
        │  State Center │
        │  （状态中心）  │
        └───────────────┘
                │
        ┌───────┴───────┐
```

**同步规则**：

| 操作来源 | 操作类型 | 对侧响应 |
|---------|---------|---------|
| 对话面板 | 用户发送消息 | 工作空间高亮相关节点/数据 |
| 对话面板 | Agent 输出结果 | 工作空间自动添加结果节点 |
| 工作空间 | 用户拖拽/点击节点 | 对话面板显示节点详情/上下文 |
| 工作空间 | 用户修改参数 | 对话面板发送确认消息 |
| 工作空间 | 用户添加注释 | 对话面板记录为对话上下文 |

**冲突解决**：以**时间戳最后写入优先** + **领域特定合并规则**（如：参数修改合并，节点删除覆盖）

### 6.4 资源调度策略

**资源预估模型**：
- 输入：数据规模 + 分析类型 + 历史数据
- 三层预估：启发式规则 + 历史回归模型 + LLM 预估
- 输出：资源推荐值（CPU/内存/时间）

**动态调度优化**：
- 队列感知：根据当前队列长度调整任务提交节奏
- 依赖优化：并行化无依赖任务，串行化有依赖任务
- 抢占策略：紧急任务可抢占低优先级任务的资源
- 回填调度：利用小任务的空隙时间执行

### 6.5 本地/云端适配

**本地模式（紧凑模式）**：Nextflow 使用 `local` executor，利用本地多核并行
**HPC 模式（弹性模式）**：Nextflow 使用 `slurm` executor，自动提交到计算节点
**云端模式（扩展模式）**：通过适配器对接 AWS Batch / GCP Life Sciences

```python
class ExecutionAdapterFactory:
    def get_adapter(self, mode: ExecutionMode) -> ExecutionAdapter:
        adapters = {
            ExecutionMode.LOCAL: LocalAdapter(),
            ExecutionMode.SLURM: SlurmAdapter(),
            ExecutionMode.AWS_BATCH: AWSBatchAdapter(),
            ExecutionMode.GCP: GCPBatchAdapter(),
        }
        return adapters[mode]
```

### 6.6 错误恢复与断点续跑

**失败分类与处理**：

| 类型 | 检测方式 | 处理策略 |
|------|---------|---------|
| 资源不足 | OOM/超时 | 自动扩容重试 |
| 代码错误 | 非零退出码 | 标记失败，通知用户 |
| 数据问题 | 输出异常 | 回溯检查输入数据 |
| 依赖缺失 | ImportError | 自动安装缺失包 |
| 节点故障 | 状态消失 | 重新提交到其他节点 |

**断点续跑机制**：
- Nextflow 天然支持：已完成的输出文件不会重新生成
- 修改参数后只重新执行受影响的下游任务
- Agent 记住断点位置，用户返回时自动恢复
- 支持"从第 N 步重新开始"的语义化操作

### 8.2 数据管理

**数据生命周期**：上传 → 验证 → 索引 → 分析 → 归档 → 清理

**版本控制策略**：
- 原始数据：不可变（Immutable），上传后锁定
- 分析结果：版本化（Versioned），每次运行生成新版本
- 中间文件：可丢弃（Ephemeral），LRU 自动清理

**命名约定**：
```
raw/sample_A_R1.fastq.gz
processed/qc/v1/sample_A_filtered.h5ad
processed/qc/v2/sample_A_filtered.h5ad  # 重新运行
```

### 8.3 分析会话管理

Session = 一次完整的分析对话 + 执行记录

**会话操作**：
- 保存：手动保存当前会话状态
- 恢复：从断点恢复分析（数据未变时）
- 重命名：给会话起有意义的名字
- 复制：基于已有会话创建新会话（换参数重跑）
- 删除：清理不再需要的会话

### 8.4 静态报告系统

报告是**可分享的分析成果**，需要兼顾**专业性**和**美观性**。

**报告内容结构**：

- methods: 方法描述（含工具、参数、代码）
- results: 结果（图表、表格、解读）
- conclusions: 结论与建议
- appendix: 完整参数、软件版本、运行时日志

**报告模板系统**（内置 + 可扩展）：

| `spatial_omics` | 空间组学报告 | 空间分析结果 |
| `experiment_design` | 实验设计报告 | Protocol 与设计 |
| `primer_design` | 引物设计报告 | 引物验证结果 |
| `drug_screening` | 药物筛选报告 | 筛选分析结果 |

**分享机制**：
- 链接分享（公开/密码保护）
- 过期时间设置
- 允许下载开关
- 查看统计

### 8.5 简单的团队权限

简化 RBAC：

| 角色 | 权限 |
|------|------|

团队工作区：

### 8.6 数据血缘与可复现性

**分析血缘图谱**：记录从原始数据到最终结果的完整链路

**可复现性保证**：
- 每个结果文件附带 JSON 格式的血缘记录

### 9.2 全局错误处理

**错误分层模型**：

| Layer | 错误类型 | 处理策略 |
|-------|---------|---------|
| 前端 | 网络断开/超时 | 自动重连 + 离线队列 |
| API 网关 | 认证失败/限流 | 明确错误码 + 友好提示 |
| Agent 核心 | LLM 调用失败 | 降级到备用模型/缓存响应 |
| Skills 运行时 | 依赖缺失/环境错误 | 自动修复/隔离失败 |
| HPC 调度 | 资源不足/节点故障 | 自动重试/扩容/通知用户 |
| 数据存储 | 文件损坏/存储满 | 校验和检测/清理策略 |

**降级策略（Graceful Degradation）**：

| 场景 | 降级行为 |
|------|---------|
| LLM 服务不可用 | 切换备用模型 → 简化响应 → 预置模板 |
| HPC 队列拥堵 | 提示等待时间 → 提供本地执行选项 |
| 文件上传失败 | 分片重试 → 断点续传 → 通知用户 |
| 可视化渲染超时 | 生成静态图 → 提供数据下载 |
| 技能执行失败 | 自动重试 → 替代技能 → 人工介入 |
| 上下文超限 | 自动压缩 → 摘要模式 → 分页加载 |

### 9.3 安全架构

**数据安全**：
- 传输加密：TLS 1.3（全链路 HTTPS/WSS）
- 存储加密：AES-256（静态数据）
- 数据隔离：项目级隔离 + 用户级访问控制
- 敏感信息：PII/PHI 数据标记 + 脱敏处理
- 审计日志：所有数据访问记录
- 备份策略：3-2-1 备份

**执行安全**：
- 沙箱隔离：Skills 在 Docker/Firecracker 中执行
- 资源限制：CPU/内存/磁盘/网络配额
- 代码审计：社区 Skills 需通过静态分析
- 网络隔离：Skills 默认无网络访问（白名单制）
- 权限最小化：执行用户无 sudo 权限
- 输入校验：所有用户输入经过 Schema 校验

**认证与授权**：
- 认证方式：OAuth 2.0 / OIDC（支持机构账号）
- 多因素认证：敏感操作需二次验证
- 会话管理：JWT + Refresh Token，自动过期
- API 密钥：程序化访问支持 scoped API key

### 9.4 测试策略

**测试金字塔**：
- 单元测试 80%：pytest + mock
- 集成测试 15%：pytest + TestContainers
- E2E 测试 5%：Playwright / Cypress

**关键测试场景**：
- Agent 核心：任务拆解正确性、Multi-Agent 协作、上下文压缩、HITL 触发
- Skills 系统：每个内置 Skill 单元测试、沙箱隔离、资源限制
- HPC 调度：适配器兼容性、断点续跑、资源预估准确性
- 前端：双轨同步、文件上传、报告渲染

**生信专项测试**：
- 金标准数据集：PBMC 3k（单细胞）、Visium 小鼠大脑（空间组学）
- 结果一致性：同一分析多次运行结果一致（允许浮点误差）
- 性能基准：质控 1万细胞 < 2分钟/4GB，聚类 1万细胞 < 5分钟/8GB

### 12.2 设计决策记录

| 决策 | 选项 | 选择 | 原因 |
|------|------|------|------|
| 部署模式 | A/B/C | C（混合） | 兼顾数据主权与协作便利 |
| MVP 策略 | A/B/C | A（Agent大脑优先） | 先确保"脑子好使"，再增强"手脚能力" |
| Multi-Agent | A/B/C | C（混合） | 兼顾效率与灵活性 |
| 交互形态 | A/B/C | C（双轨） | 对话灵活 + 可视化直观 |
| 工作流引擎 | Nextflow/Snakemake | Nextflow | 首选，生态更活跃 |
| 前端框架 | React/Vue/Svelte | React | 生态最成熟，ReactFlow 完善 |
| 协作模式 | 实时协作/静态分享 | 静态分享 | 单用户为主，降低复杂度 |
| 部署简化 | Docker/Conda/pip | pip + Conda | 降低使用门槛 |
