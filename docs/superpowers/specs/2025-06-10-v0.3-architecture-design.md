## 一、需求与参考信息的批判性评估

### 1.1 新增功能需求的评估

| 需求 | 合理性 | 必要性 | 风险 | 判断 |
|------|--------|--------|------|------|
| **Plan模式** | ★★★★★ | **极高** | 过度设计导致延迟 | **立即实施**。当前硬编码pipeline是最大瓶颈，没有Plan模式就无法处理非标准分析流程 |
| **分析结果主动解读** | ★★★★★ | **极高** | LLM幻觉、token成本高 | **立即实施**。这是"主动智能分析伙伴"定位的核心体现，不做则与竞品无差异 |
| **持久化工作目录** | ★★★★★ | **极高** | 数据安全、磁盘管理 | **立即实施**。基础功能缺失，当前tmp_path导致中间结果无法复用 |

**结论**：三个需求全部合理且必要，不存在"过度需求"。当前v0.2.0的硬编码pipeline和临时工作目录已经成为用户体验的最大瓶颈。

### 1.2 参考项目的批判性评估

#### Ruflo（https://github.com/ruvnet/ruflo）

| 能力 | 实际评估 | 可取之处 | 弃用原因 |
|------|---------|---------|---------|
| **Swarm协调** | 100+ Agent拓扑、Raft共识、Gossip协议、Byzantine容错 | 理念先进 | **对当前单机/小团队部署过度工程化**。科研分析不需要分布式共识 |
| **GOAP A* Planner** | 目标→动作→前置条件→效果→A*搜索 | 规划方法论优秀 | **生物信息学的前置条件/效果难以精确定义**（如"数据质量合格"无法形式化） |
| **ruflo-workflows** | 可复用多步骤任务模板 | **Dynamic Workflows概念优秀** | 插件化实现过重 |
| **SONA自学习** | 神经模式学习、轨迹学习 | 理念先进 | 需要大量数据，当前规模不适用 |
| **Federation** | 跨机器零信任通信 | 安全理念好 | 完全不需要 |

**Ruflo的实质**：一个面向企业级多Agent协作的重量级框架。其Swarm协调和Federation对科研单细胞分析场景是**大炮打蚊子**。

**我们取什么**：
- ✅ **GOAP的规划思想** → 改造为"DAG约束下的LLM动态规划"（见PlanEngine设计）
- ✅ **Dynamic Workflows** → 可复用分析模板（workflow templates）
- ✅ **Hooks自动路由** → TurnRunner中增加plan/interpretation钩子点

#### SkillDAG（https://github.com/Ericbai06/SkillDAG）

| 能力 | 实际评估 | 可取之处 | 弃用原因 |
|------|---------|---------|---------|
| **DAG关系建模** | 依赖/冲突/专业化/重复四种边类型 | **核心思想极其重要** | — |
| **自演化图边注册** | agent执行后自动propose/edit边 | 理念先进 | **过于复杂，收敛慢**。当前77个技能规模，手动维护DAG边更实际 |
| **SkillsBench Benchmark** | 200任务评估 | 学术严谨 | 不需要 |

**SkillDAG的实质**：一篇顶级AI论文的代码实现，面向"大规模技能库（1000+）的自动选择"。

**我们取什么**：
- ✅ **DAG拓扑推断思想** → 技能间结构化关系取代纯相似度匹配
- ✅ **边的类型化设计** → depends_on / conflicts_with / specializes / followed_by
- ❌ **自演化机制** → 简化为SKILL.md optional字段，人工维护+可选自动推断

**关键洞察**：SkillDAG的核心价值不是"自演化"，而是**把技能选择从"相似度匹配"升级为"结构化推理"**。这正是PlanEngine需要的底层能力。

#### Odysseus（https://github.com/pewdiepie-archdaemon/odysseus）

| 能力 | 实际评估 | 可取之处 | 弃用原因 |
|------|---------|---------|---------|
| **Agent Loop模块化** | chat_processor → agent_loop → agent_tools → memory | **模块化设计优秀** | — |
| **持久化记忆** | ChromaDB + fastembed | 零运维向量存储 | 已用sqlite-vec替代 |
| **MCP集成** | 自动注册MCP服务器 | 集成模式可参考 | 已部分实现 |
| **Email/Calendar** | IMAP/SMTP/CalDAV | — | **完全无关** |
| **PWA** | 可安装Web应用 | — | **不需要** |

**Odysseus的实质**：一个功能全面的"本地ChatGPT替代品"，面向个人生产力。

**我们取什么**：
- ✅ **Agent Loop的模块化结构** → 将TurnRunner进一步拆分为ContextAssembler / Planner / Executor / Interpreter
- ✅ **本地优先设计哲学** → WorkspaceManager的设计参考
- ❌ **Email/Calendar/PWA** → 完全裁剪

#### CowAgent（https://github.com/zhayujie/CowAgent）

| 能力 | 实际评估 | 可取之处 | 弃用原因 |
|------|---------|---------|---------|
| **三层记忆架构** | context → daily → core | **架构清晰** | — |
| **Deep Dream蒸馏** | 夜间自动蒸馏记忆为长期知识 | 自动化程度高 | 当前可以用显式触发替代 |
| **Personal Knowledge Base** | 按主题组织的Markdown wiki + 知识图谱 | **核心概念优秀** | 知识图谱可视化可简化 |
| **自进化** | 自动回顾对话改进skill | 理念先进 | 当前规模下显式A/B测试已足够（已有evolution.py） |
| **多频道** | Web/WeChat/Feishu/... | — | **不需要** |

**CowAgent的实质**：一个面向个人用户的超级AI助手，前身是chatgpt-on-wechat。

**我们取什么**：
- ✅ **三层记忆架构** → 与现有WorkingMemory + SemanticMemory + ExperimentLogger对齐升级
- ✅ **Personal Knowledge Base** → UserProfile模块的设计参考
- ❌ **多频道集成** → 完全裁剪
- ❌ **Deep Dream夜间蒸馏** → 简化为显式触发（用户或系统主动请求总结）

### 1.3 评估总结

**四个参考项目的核心精华**：
1. **Ruflo**：规划思想（GOAP）+ 动态工作流模板 + 自动路由
2. **SkillDAG**：DAG结构化关系建模（不是自演化）
3. **Odysseus**：模块化Agent Loop + 本地优先
4. **CowAgent**：三层记忆 + 个人知识库

**我们明确弃用的**：
- ❌ Swarm分布式协调（Raft/Gossip/Byzantine）
- ❌ Federation跨机器通信
- ❌ Email/Calendar/PWA
- ❌ 完整DAG自演化机制
- ❌ 多频道IM集成
- ❌ SONA神经模式学习

## 三、核心模块详细设计

### 3.1 PlanEngine（计划引擎）

**设计原则**：不是GOAP（前置条件/效果难定义），也不是纯LLM规划（容易幻觉），而是**"DAG结构化约束 + LLM动态规划"的混合方案**。

#### 3.1.1 核心算法

```python
class PlanEngine:
    """
    基于SkillDAG的动态计划生成引擎。
    
    工作流程：
    1. 接收用户意图 + 当前上下文
    2. SkillDAG.search() → 找到相关技能子图（向量匹配 + 边邻居 + 冲突信号）
    3. LLM在子图约束下生成执行计划
    4. 输出TaskTree
    """
    
    async def plan(
        self,
        intent: UserIntent,
        context: PlanContext,
    ) -> TaskTree:
        # Step 1: 候选技能检索
        candidates = self.skill_dag.search(
            query=intent.to_natural_language(),
            top_k=15,
            include_neighbors=True,  # 包含DAG边邻居
            exclude_conflicts=True,   # 排除冲突技能
        )
        
        # Step 2: 子图构建
        subgraph = self.skill_dag.extract_subgraph(candidates)
        
        # Step 3: LLM规划（在DAG约束下）
        prompt = self._build_planning_prompt(intent, subgraph, context)
        plan = await self.llm.generate_structured(prompt, schema=TaskTreeSchema)
        
        # Step 4: 验证（DAG约束检查）
        validated = self._validate_against_dag(plan, subgraph)
        
        # Step 5: 存储计划模式供后续学习
        await self.semantic_memory.add(
            text=f"Plan for {intent.analysis_type}: {plan.summary}",
            memory_type="plan",
            metadata={"intent": intent.model_dump(), "plan": plan.model_dump()},
        )
        
        return validated
```

#### 3.1.2 SkillDAG数据结构

```python
@dataclass
class SkillDAGEdge:
    """技能间的DAG边，用于结构化推理。"""
    from_skill: str
    to_skill: str
    edge_type: Literal["depends_on", "conflicts_with", "specializes", "followed_by", "alternative_to"]
    confidence: float = 1.0  # 人工维护=1.0，自动推断<1.0
    context: str = ""  # 为什么有这样的关系

class SkillDAG:
    """
    技能有向图，取代纯语义搜索。
    
    核心能力：
    - search(): 向量匹配 + 图邻居扩展
    - get_dependencies(): 获取技能的所有依赖链
    - get_conflicts(): 获取技能的所有冲突
    - get_recommendations(): 基于followed_by边推荐后续技能
    """
```

#### 3.1.3 SKILL.md扩展

```yaml
---
name: scanpy_cluster
description: Compute neighbors and UMAP embedding for single-cell data
whenToUse: After QC and dimensionality reduction, before cell annotation
keywords: [single-cell, clustering, UMAP, neighbors]

# 新增：DAG边信息（可选，用于PlanEngine）
dag_edges:
  depends_on: [scanpy_qc, scanpy_pca]  # 必须前置的技能
  conflicts_with: [seurat_cluster]  # 互斥技能（分析路径二选一）
  specializes: [generic_clustering]  # 更专业的版本
  followed_by: [scanpy_annotation, scanpy_de]  # 常见后续
  alternative_to: [leiden_clustering, louvain_clustering]  # 替代方案
---
```

**为什么`followed_by`很重要**：
- `depends_on`是**硬性约束**（必须先QC再聚类）
- `followed_by`是**软性推荐**（聚类后通常做注释，但也可以直接做DE）
- PlanEngine在硬性约束满足后，用`followed_by`生成推荐路径

#### 3.1.4 与Ruflo GOAP的对比

| 维度 | Ruflo GOAP | HomomicsLab PlanEngine |
|------|-----------|----------------------|
| 规划方式 | 状态空间A*搜索 | DAG约束下的LLM规划 |
| 前置条件 | 精确定义（布尔表达式） | 隐式约束（depends_on边） |
| 效果 | 精确定义（状态变更） | 隐式（followed_by推荐） |
| 适用场景 | 游戏AI、软件工程 | 数据驱动的科研分析 |
| 优势 | 确定性、可解释 | 灵活、可处理模糊约束 |

**判断**：GOAP对于生物信息学不适用，因为"数据质量合格"、"批次效应已校正"这类条件无法精确定义为布尔表达式。我们的DAG+LLM混合方案是更适合的选择。

### 3.2 InterpretationEngine（解读引擎）

**设计原则**：不是每个步骤都解读（太啰嗦），而是**phase级别解读** + **异常触发解读** + **用户主动请求解读**。

#### 3.2.1 三层解读

```python
class InterpretationEngine:
    """
    分析结果主动解读与推荐引擎。
    
    三层解读：
    1. Phase Interpretation: 每个phase（preprocessing/analysis/reporting）结束时
    2. Anomaly-triggered: 检测到异常时（如QC过滤率>30%）
    3. On-demand: 用户主动请求解读
    """
    
    async def interpret_task_result(
        self,
        task: TaskNode,
        result: Dict[str, Any],
        context: InterpretationContext,
    ) -> InterpretationResult:
        
        # Layer 1: 数据质量评估
        quality_assessment = self._assess_quality(task, result)
        
        # Layer 2: 语义解读（调用LLM）
        interpretation = await self._generate_interpretation(
            task=task,
            result=result,
            quality=quality_assessment,
            history=await self.semantic_memory.search(
                query=f"similar to {task.name} on {context.data_description}",
                top_k=3,
            ),
        )
        
        # Layer 3: 推荐生成
        recommendations = self._generate_recommendations(
            task=task,
            result=result,
            skill_dag=self.skill_dag,
        )
        
        return InterpretationResult(
            summary=interpretation.summary,
            key_findings=interpretation.findings,
            quality_assessment=quality_assessment,
            recommendations=recommendations,
            confidence=interpretation.confidence,
        )
```

#### 3.2.2 解读触发条件

```python
class InterpretationTrigger:
    """解读触发器，决定何时进行解读。"""
    
    # 1. Phase结束（硬性触发）
    PHASE_COMPLETE = "phase_complete"
    
    # 2. 异常检测（条件触发）
    ANOMALY_DETECTED = "anomaly_detected"
    # 条件示例：
    # - QC过滤率 > 20%
    # - 聚类数与预期差异 > 2x
    # - 批次效应p值 < 0.05
    
    # 3. HITL checkpoint（硬性触发）
    HITL_CHECKPOINT = "hitl_checkpoint"
    
    # 4. 用户主动请求（用户触发）
    USER_REQUEST = "user_request"
```

#### 3.2.3 推荐生成算法

```python
def _generate_recommendations(self, task, result, skill_dag) -> List[Recommendation]:
    """基于SkillDAG和当前结果状态生成推荐。"""
    recommendations = []
    
    # 1. 基于followed_by边的直接推荐
    for skill_id in skill_dag.get_followed_by(task.skills_required[0]):
        recommendations.append(Recommendation(
            type="next_step",
            skill_id=skill_id,
            reason=f"通常在进行{task.name}后会执行此步骤",
            confidence=0.8,
        ))
    
    # 2. 基于结果异常的修复推荐
    if result.get("qc_filter_rate", 0) > 0.3:
        recommendations.append(Recommendation(
            type="caution",
            skill_id="scanpy_qc_advanced",
            reason="过滤率过高（>30%），建议检查QC参数或数据质量",
            confidence=0.9,
        ))
    
    # 3. 基于历史模式的学习推荐
    similar_plans = await self.semantic_memory.search(
        query=f"after {task.name} what to do next",
        memory_type="plan",
        top_k=3,
    )
    for plan in similar_plans:
        recommendations.append(Recommendation(
            type="learned_pattern",
            skill_id=plan.metadata.get("next_skill"),
            reason="基于历史分析模式",
            confidence=plan.score,
        ))
    
    # 去重+排序
    return self._deduplicate_and_rank(recommendations)
```

### 3.4 Skill渐进披露系统

**设计确认**：CSV文件中定义的L1/L2/L3三层架构是**正确且合理的**。但需要补充几个关键点。

#### 3.4.1 三层架构确认

| 层级 | 内容 | 加载时机 | Token成本 | 控制权 |
|------|------|---------|-----------|--------|
| **L1** | YAML Frontmatter + DAG边信息 | Session启动时 | ~50-100 / skill | 系统层 |
| **L2** | SKILL.md Body（完整指令、示例） | PlanEngine选择技能时 | ~3K-5K / skill | 系统层 |
| **L3** | scripts/、references/、assets/ | 执行时按需 | 仅加载被引用 | 系统层 |

**关键改进**：
1. **披露控制权在系统层，不在模型层**：ContextAssembler根据当前执行状态自动组装上下文，模型不应自己决定加载什么
2. **L1必须包含DAG边信息**：PlanEngine依赖这些信息做结构化推理
3. **L2的加载粒度**：不是加载单个技能，而是加载**计划涉及的所有技能**（通常2-5个）

#### 3.4.2 ContextAssembler实现

```python
class ContextAssembler:
    """
    根据当前执行状态，自动组装LLM上下文。
    
    核心职责：
    - 管理L1/L2/L3的加载和组装
    - 确保上下文不超过token预算
    - 维护技能上下文的cache
    """
    
    def __init__(
        self,
        skill_registry: SkillRegistry,
        skill_dag: SkillDAG,
        token_budget: int = 32000,
    ):
        self.skill_registry = skill_registry
        self.skill_dag = skill_dag
        self.token_budget = token_budget
        self._l1_cache: Dict[str, str] = {}  # skill_id -> L1上下文
        self._l2_cache: Dict[str, str] = {}  # skill_id -> L2上下文
    
    def assemble_for_planning(self) -> str:
        """为PlanEngine组装L1上下文（所有技能的YAML frontmatter + DAG边）。"""
        # L1常驻上下文，Session启动时一次性构建
        if not self._l1_cache:
            for skill in self.skill_registry.list_all():
                self._l1_cache[skill.id] = self._build_l1_context(skill)
        return "\n".join(self._l1_cache.values())
    
    def assemble_for_execution(self, skill_ids: List[str]) -> str:
        """为执行组装L2上下文（选中技能的完整指令）。"""
        contexts = []
        for skill_id in skill_ids:
            if skill_id not in self._l2_cache:
                skill = self.skill_registry.get(skill_id)
                self._l2_cache[skill_id] = self._build_l2_context(skill)
            contexts.append(self._l2_cache[skill_id])
        return "\n".join(contexts)
    
    def _build_l1_context(self, skill: SkillDefinition) -> str:
        """构建L1上下文（精简描述 + DAG边）。"""
        lines = [
            f"<skill id='{skill.id}'>",
            f"  <name>{skill.name}</name>",
            f"  <description>{skill.description}</description>",
            f"  <whenToUse>{skill.metadata.get('whenToUse', '')}</whenToUse>",
        ]
        
        # 新增：DAG边信息
        dag = skill.metadata.get("dag_edges", {})
        if dag:
            lines.append("  <dag_edges>")
            for edge_type, targets in dag.items():
                lines.append(f"    <{edge_type}>{', '.join(targets)}</{edge_type}>")
            lines.append("  </dag_edges>")
        
        lines.append("</skill>")
        return "\n".join(lines)
```

#### 3.4.3 Token预算管理

```
总Token预算（假设32K上下文窗口）：
├── 系统prompt: ~2K
├── 工作记忆（最近10轮对话）: ~5K
├── L1技能目录（77技能 × 100 tokens）: ~8K
├── L2选中技能（3技能 × 4K tokens）: ~12K
├── 当前任务上下文: ~3K
└── 预留: ~2K
```

**如果技能数量增长到200+**：
- L1成本将增长到~20K，超过预算
- **解决方案**：L1分级加载
  - 核心技能（20个builtin + 常用external）：常驻L1
  - 其他技能：按需通过语义搜索加载到L1

### 3.5 Agent体系重构：Skill/Tool/Plugin定位

#### 3.5.1 当前问题

当前架构中：
- `BaseAgent`是抽象类，`BioinfoAgent`/`VizAgent`是硬编码子类
- Agent通过`skill_executor`运行skill
- AgentRegistry注册的是**Agent实例**
- 没有Tool的概念，MCP工具直接暴露在mcp/client.py中

**问题**：
1. Agent类与Role绑定过紧，新增一个"Reviewer"角色需要新建一个类
2. Tool和Skill混为一谈，导致MCP工具无法被Agent统一调度
3. Plugin没有定义，扩展方式不明确

#### 3.5.2 新定位

#### 3.5.3 ToolRegistry设计

```python
class ToolDefinition(BaseModel):
    """原子能力的定义。"""
    name: str
    description: str
    input_schema: Dict[str, Any]  # JSON Schema
    output_schema: Dict[str, Any]
    handler: Callable  # 执行函数
    source: Literal["mcp", "builtin", "plugin"]  # 来源
    metadata: Dict[str, Any] = Field(default_factory=dict)

class ToolRegistry:
    """原子能力注册表，与SkillRegistry互补。"""
    
    def __init__(self):
        self._tools: Dict[str, ToolDefinition] = {}
    
    def register_from_mcp(self, mcp_client: BioMCPClient) -> None:
        """从MCP服务器动态注册工具。"""
        for tool in mcp_client.list_tools():
            self.register(ToolDefinition(
                name=tool.name,
                description=tool.description,
                input_schema=tool.input_schema,
                handler=lambda inputs, t=tool: mcp_client.call_tool(t.name, inputs),
                source="mcp",
            ))
    
    def register_builtin(self, name: str, handler: Callable, **kwargs) -> None:
        """注册内置工具。"""
        self.register(ToolDefinition(name=name, handler=handler, source="builtin", **kwargs))
    
    def get_for_role(self, role: AgentRole) -> List[ToolDefinition]:
        """获取某个Agent角色可用的工具。"""
        # 每个角色有默认工具集，可配置
        role_tools = {
            AgentRole.BIOINFO: ["file_read", "file_write", "shell_exec", "web_search"],
            AgentRole.VIZ: ["file_read", "plotly_render", "matplotlib_render"],
            AgentRole.INTERPRETER: ["memory_search", "semantic_search", "web_search"],
        }
        tool_names = role_tools.get(role, ["file_read"])
        return [self._tools[name] for name in tool_names if name in self._tools]
```

#### 3.5.4 AgentCore：动态角色注入

```python
class AgentRole(str, Enum):
    """Agent角色定义。新增角色无需修改代码，只需添加配置。"""
    PLANNER = "planner"       # 任务分解与计划生成
    BIOINFO = "bioinfo"       # 生物信息学分析
    VIZ = "viz"               # 可视化
    INTERPRETER = "interpreter"  # 结果解读
    CRITIC = "critic"         # 质量检查（新增）
    REPORTER = "reporter"     # 报告生成

class AgentCore:
    """
    统一Agent核心，通过角色注入实现多专家协作。
    
    与v0.2硬编码Agent的区别：
    - v0.2: BioinfoAgent(类) → 只能做生物信息学
    - v0.3: AgentCore(角色=bioinfo) → 同一实例可切换角色
    """
    
    def __init__(
        self,
        llm_router: LLMRouter,
        memory: SemanticMemory,
        skill_registry: SkillRegistry,
        tool_registry: ToolRegistry,
        workspace_manager: WorkspaceManager,
    ):
        self.llm_router = llm_router
        self.memory = memory
        self.skill_registry = skill_registry
        self.tool_registry = tool_registry
        self.workspace_manager = workspace_manager
    
    async def execute(
        self,
        task: TaskNode,
        role: AgentRole,
        context: Dict[str, Any],
    ) -> Dict[str, Any]:
        """执行时动态注入角色上下文。"""
        
        # 1. 构建角色系统prompt
        system_prompt = self._build_role_prompt(role)
        
        # 2. 获取该角色可用的capabilities
        available_skills = self.skill_registry.get_for_role(role)
        available_tools = self.tool_registry.get_for_role(role)
        
        # 3. 构建完整上下文（L1/L2披露）
        full_context = self._assemble_context(
            task=task,
            role=role,
            skills=available_skills,
            tools=available_tools,
            memory=await self._retrieve_relevant_memory(task),
        )
        
        # 4. LLM规划执行（ReAct/Tool-use模式）
        response = await self.llm_router.generate(
            system=system_prompt,
            messages=full_context,
            tools=[t.model_dump() for t in available_tools],
        )
        
        # 5. 执行Tool/Skill调用
        result = await self._execute_tool_calls(response.tool_calls)
        
        # 6. 存储记忆
        await self.memory.add(...)
        
        return result
```

## 四、关键设计取舍

### 4.1 取舍决策表

| # | 取舍点 | 选项A | 选项B | **选择** | 原因 |
|---|--------|-------|-------|---------|------|
| 1 | 规划算法 | GOAP A* | 纯LLM | **DAG+LLM混合** | GOAP前置条件在生物信息学中难定义；纯LLM易幻觉 |
| 2 | Agent协调 | Swarm分布式 | 单Agent | **单Agent+动态角色** | 单机场景不需要分布式共识 |
| 3 | DAG边维护 | 自动自演化 | 手动维护 | **手动维护+可选自动推断** | 77技能规模手动更可控 |
| 4 | 记忆数据库 | ChromaDB | sqlite-vec | **继续sqlite-vec** | 当前规模足够，零运维 |
| 5 | 工作目录 | 全局共享 | 项目隔离 | **项目隔离** | 防止项目间数据污染 |
| 6 | 解读频率 | 每步解读 | Phase级解读 | **Phase级+异常触发** | 每步解读太啰嗦，token成本高 |
| 7 | L1加载策略 | 全部加载 | 按需加载 | **核心常驻+按需扩展** | 预算控制 |
| 8 | Tool vs Skill | 统一概念 | 严格区分 | **严格区分** | 不同粒度、不同生命周期 |

### 4.2 明确不做的（避免过度工程化）

1. **❌ 分布式Swarm协调** — Raft/Gossip/Byzantine对单机科研无意义
2. **❌ 跨机器Federation** — 当前无需多机协作
3. **❌ 完整的DAG自演化** — 论文级机制，收敛慢
4. **❌ 夜间自动Deep Dream** — 简化为显式触发
5. **❌ 知识图谱可视化** — Markdown wiki足够，可视化可延后
6. **❌ 多LLM后端路由** — 可先实现接口，延后多供应商支持
7. **❌ 自动A/B测试进化** — 已有evolution.py，但暂不自动触发

## 八、总结

### 8.1 设计哲学

**"取所长，不拼凑"的具体实践**：

1. **Ruflo的GOAP → DAG+LLM混合**：不实现A*状态搜索，而是借用"目标分解"思想，用DAG约束解决幻觉问题
2. **Ruflo的Swarm → 动态角色**：不实现100+ Agent和分布式共识，而是将"多专家协作"简化为"同一Agent切换system prompt"
3. **SkillDAG的自演化 → 手动边维护**：不实现完整的边注册和演化机制，而是将DAG边作为SKILL.md的optional字段
4. **Odysseus的完整平台 → 裁剪**：只采纳agent_loop和持久化记忆设计，不引入Email/Calendar/PWA
5. **CowAgent的多频道 → 专注科研**：只采纳三层记忆和个人知识库，不做IM集成

### 8.2 核心价值主张

v0.3.0的核心升级不是"更多功能"，而是**"从被动执行到主动智能"的质变**：

- **PlanEngine**：Agent不再只是执行用户命令，而是能理解意图、自主规划分析路径
- **InterpretationEngine**：Agent不再只是返回结果，而是能解读生物学意义、推荐下一步
- **WorkspaceManager**：Agent不再只是临时处理数据，而是能管理完整的分析生命周期

这三个模块的协同，将使HomomicsLab从一个"生物信息学工具执行平台"真正进化为一个"生物信息学智能分析伙伴"。
