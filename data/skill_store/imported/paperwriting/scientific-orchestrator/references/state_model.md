# State Model Specification

## Project State File

**Filename:** `.scientific-project.json`
**Location:** Working directory of the project (same directory where research files are stored)
**Format:** JSON
**Created by:** `scripts/state_manager.py --init`
**Updated by:** `scripts/state_manager.py --complete-stage <stage>` or `--add-artifact <path>`

## Schema Definition

```json
{
  "$schema": "scientific-orchestrator/v1",
  "project_id": "string (uuid)",
  "created_at": "string (ISO 8601)",
  "updated_at": "string (ISO 8601)",
  "version": "string (semver)",
  "current_stage": "enum string",
  "completed_stages": ["array of enum strings"],
  "active_workflow": "string (workflow_id)",
  "workflows": {
    "<workflow_id>": {
      "status": "enum: not_started | in_progress | paused | completed",
      "current_step": "integer (1-based)",
      "total_steps": "integer",
      "started_at": "string (ISO 8601)",
      "completed_at": "string (ISO 8601, optional)"
    }
  },
  "artifacts": {
    "<stage_name>": ["array of relative file paths"]
  },
  "metadata": {
    "title": "string",
    "description": "string",
    "target_journal": "string",
    "target_conference": "string",
    "funding_agency": "string",
    "deadline": "string (ISO 8601 date)",
    "research_field": "string",
    "keywords": ["array of strings"],
    "notes": "string"
  }
}
```

## Stage Enumeration

Valid values for `current_stage` and items in `completed_stages`:

| Value | Display Name | Corresponding Skill |
|-------|-------------|---------------------|
| `ideation` | Research Ideation | scientific-ideation |
| `literature` | Literature Review | scientific-literature-review |
| `design` | Research Design | scientific-research-design |
| `analysis` | Data Analysis | (external tools) |
| `visualization` | Figure Visualization | scientific-visualization |
| `illustration` | Scientific Illustration | scientific-illustration |
| `manuscript` | Manuscript Writing | scientific-manuscript |
| `review` | Peer Review | scientific-peer-review |
| `communication` | Scientific Communication | scientific-communication |
| `grant` | Grant Writing | scientific-grant-writing |

## Artifact Detection Rules

The `scripts/detect_state.py` tool uses these heuristics to infer stage completion:

### Stage: ideation
**Markers:**
- `research_question.md` exists
- `risk_assessment.md` exists
- `decision_tree.md` exists
- `hypothesis.md` exists

**Confidence:** High if 2+ markers found

### Stage: literature
**Markers:**
- `literature_review.md` exists
- `knowledge_gaps.md` exists
- `citations.bib` or `.bib` file exists with >10 entries
- `references/` directory with >5 files

**Confidence:** High if `literature_review.md` exists; Medium if only `.bib` found

### Stage: design
**Markers:**
- `experimental_design.md` exists
- `hypothesis_framework.md` exists
- `statistical_plan.md` exists
- `protocol.md` exists

**Confidence:** High if `experimental_design.md` exists

### Stage: analysis
**Markers:**
- `data/` directory exists with data files
- `results/` directory exists
- `*.csv`, `*.xlsx`, `*.h5`, `*.RDS` data files present
- `analysis.R`, `analysis.py`, `*.ipynb` scripts present

**Confidence:** Medium (cannot distinguish between raw and analyzed data)

### Stage: visualization
**Markers:**
- `figures/` directory exists
- `*.png`, `*.pdf`, `*.tiff`, `*.svg` files in `figures/`
- `figure_captions.md` exists

**Confidence:** High if publication-quality formats (PDF/TIFF/SVG) present

### Stage: illustration
**Markers:**
- `schematics/` directory exists
- `diagrams/` directory exists
- `*.png` schematic files present
- Files generated by `generate_schematic.py` detected

**Confidence:** High if schematics directory exists

### Stage: manuscript
**Markers:**
- `manuscript.md` exists
- `manuscript.docx` exists
- `*.tex` LaTeX files present
- `abstract.md` exists
- `introduction.md`, `methods.md`, `results.md`, `discussion.md` exist

**Confidence:** High if `manuscript.md` or `.tex` found

### Stage: review
**Markers:**
- `review_report.md` exists
- `reviewer_response.md` exists
- `revision_notes.md` exists
- Tracked changes in document files

**Confidence:** High if `review_report.md` or `reviewer_response.md` found

### Stage: communication
**Markers:**
- `*.pptx` presentation files exist
- `*.pdf` poster files exist
- `poster.tex` or `slides.tex` exists
- `website/` or `docs/` directory exists

**Confidence:** High if PPTX or poster PDF found

### Stage: grant
**Markers:**
- `proposal.pdf` exists
- `proposal.md` exists
- `budget.xlsx` or `budget.csv` exists
- `timeline.gantt` or `timeline.md` exists
- `biosketch.md` exists
- Specific agency files: `specific_aims.md`, `project_summary.md`, `立项依据.md`

**Confidence:** High if `proposal.*` or agency-specific files found

## State Transition Rules

### Allowed Transitions

From any stage, the orchestrator can transition to:
- The next stage in the active workflow
- Any stage that has already been completed (for iteration)
- The `analysis` stage (for returning to data work)
- Any stage with an explicit user request

### Transition Validation

`scripts/workflow_runner.py` validates transitions:

| From | To | Valid? | Notes |
|------|-----|--------|-------|
| ideation | literature | Yes | Natural flow |
| ideation | manuscript | Warning | Missing literature and design stages |
| visualization | manuscript | Yes | Data-to-paper workflow |
| manuscript | communication | Warning | Should include peer-review first |
| any | any | Yes (override) | User can always force a transition |

## Multi-Project Support

A single working directory may contain multiple projects. In this case:

1. Each project gets its own state file: `.scientific-project-<id>.json`
2. The `detect_state.py` script lists all detected projects
3. The `state_manager.py` accepts `--project-id` to target a specific project

## Recovery Scenarios

### Scenario: State File Corrupted

**Detection:** JSON parse error when reading `.scientific-project.json`
**Recovery:**
1. Rename corrupted file to `.scientific-project.json.bak`
2. Run `detect_state.py` to reconstruct state from artifacts
3. Initialize new state with reconstructed information
4. Prompt user to verify reconstructed state

### Scenario: User Works Outside Skill System

**Detection:** Files created manually without state updates
**Recovery:**
1. Run `detect_state.py` to detect new artifacts
2. Offer to update state file with discovered artifacts
3. Ask user to confirm stage advancement if artifacts suggest it

### Scenario: Workflow Switch Mid-Project

**Example:** User was working on `paper_pipeline` but now needs a `grant_pipeline`
**Recovery:**
1. Mark current workflow as `paused`
2. Initialize new workflow
3. Copy relevant completed stages to new workflow
4. Allow parallel tracking (both workflows appear in state)
